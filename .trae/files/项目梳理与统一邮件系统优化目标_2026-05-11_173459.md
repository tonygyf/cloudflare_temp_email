# cloudflare_temp_email 项目梳理与统一邮件系统优化目标

- 生成时间: `2026-05-11 17:34:59 +08:00`
- 目标: 将当前“临时邮箱系统”升级为“Cloudflare 多项目统一邮件后端”，服务于验证码、事务通知、收件回调、统一管理。

## 1. 项目结构识别（按功能域）

### 1.1 核心运行面
- `worker/`: Cloudflare Workers 后端主服务（Hono）。
  - 路由层: `src/worker.ts`
  - 管理 API: `src/admin_api/*`
  - 邮箱 API: `src/mails_api/*`
  - 用户 API: `src/user_api/*`
  - 收信流程: `src/email/*`
- `frontend/`: Vue 3 管理与邮箱 UI（部署到 Cloudflare Pages）。
- `pages/functions/_middleware.js`: Pages Function 中间件，将 `/api`、`/admin` 等请求转发到 Worker。

### 1.2 邮件能力扩展面
- `mail-parser-wasm/`: Rust WASM 邮件解析模块（高性能解析）。
- `smtp_proxy_server/`: Python SMTP/IMAP 代理服务（给邮件客户端或外部系统接入）。
- `skills/cf-temp-mail-agent-mail/`: Agent 邮箱调用能力（对 AI Agent 场景友好）。

### 1.3 数据与配置面
- `db/`: D1 Schema 与演进 SQL（已包含 2026 年补丁）。
- `worker/wrangler.toml.template`: Worker 主要绑定与变量模板（D1/KV/AI/服务绑定/限流等）。
- `frontend/.env.example`: 前端入口配置（如 `VITE_API_BASE`）。

### 1.4 质量与发布面
- `e2e/`: API/Browser/SMTP Proxy 自动化测试。
- `.github/workflows/`: Worker、Frontend、Docs、E2E 自动化发布与验证流水线。
- `vitepress-docs/`: 中文/英文文档体系（部署、功能、API、运维）。

## 2. 当前部署要点（识别结果）

### 2.1 Cloudflare 关键组件
- Worker: 承载后端 API + Email Worker 处理入口。
- D1: 邮件、地址、用户等核心结构化数据。
- Pages: 前端页面托管，配合 Function 转发 API 到 Worker。
- Email Routing: 将目标域名邮件路由到 Worker 收件。
- 可选绑定:
  - Workers AI（AI 验证码/链接提取）
  - KV（验证码等临时状态）
  - R2/S3（附件持久化）
  - RateLimit binding（防滥用）

### 2.2 路由与鉴权现状（对统一后端很关键）
- 已分层鉴权:
  - `/api/*`: 地址 JWT
  - `/user_api/*`: 用户 token
  - `/admin/*`: 管理密码
- 已具备私有化能力:
  - 全局访问密码
  - 管理后台密码
  - 可扩展 IP 黑白名单/角色域名隔离

### 2.3 发信与收信现状
- 发信: 支持 SMTP / Resend / MailChannels + DKIM 相关配置能力。
- 收信: Email Worker + 邮件解析 + 存储 + 转发 + webhook。
- 已有附加能力: Telegram 推送、AI 提取、代理协议接入（IMAP/SMTP）。

## 3. 面向“统一邮件后端”的优化目标（分阶段）

## 3.1 第一阶段（高优先级，先打通业务）
- 目标: 让“其他 CF Worker/应用”可稳定调用当前系统完成验证码和事务邮件。
- 改造项:
  - 新增“应用级 API Key”模型（按应用发 key，而不是共用管理员凭据）。
  - 权限分级:
    - 仅允许模板发信
    - 限制可用发件域与地址前缀
    - 可选 IP 白名单
  - 新增“模板发送 API”:
    - 模板表 `email_templates`
    - 变量渲染（如 `{{code}}`、`{{username}}`）
  - 加入发送配额与速率限制:
    - 按应用 key、IP、收件人域名多维限流
    - 发送日志与失败原因留存

## 3.2 第二阶段（收件回流与自动化）
- 目标: 把收件能力变成“其他系统可订阅的数据流”。
- 改造项:
  - 规则化 Webhook:
    - 按前缀/域名/关键词路由到不同下游 Worker
    - HMAC 签名校验，防伪造回调
  - 验证码提取增强:
    - AI + 规则双引擎
    - 命中后主动回调业务系统（降低轮询成本）
  - 地址隔离策略:
    - 应用前缀命名规范（`app1-xxx@domain`）
    - 关联应用标签，实现跨项目可观测

## 3.3 第三阶段（平台化与运维）
- 目标: 从“功能可用”升级为“长期稳定的邮件基础设施”。
- 改造项:
  - 审计与可观测:
    - 关键操作审计日志
    - 成功率、退信率、提取成功率仪表盘
  - 安全加强:
    - Cloudflare Access/Turnstile 全面保护
    - 高危接口二次验证
  - 数据生命周期:
    - D1 归档策略
    - R2/S3 附件保留与清理策略
    - 自动备份与恢复演练

## 5. 风险与前置依赖

- 发信通道合规与投递信誉（SPF/DKIM/DMARC）需先校准。
- 跨项目复用时，必须避免“单 key 全权限”模式。
- webhook 回调若无签名和重放保护，会有安全风险。
- D1 容量和查询模式需要随发送量增长提前优化索引与清理策略。

3. 保持与现有 `admin_api`、`mails_api` 目录风格一致，避免大规模重构。
4. 在 `e2e/tests/api/` 增加 key 权限与模板发送回归测试。
